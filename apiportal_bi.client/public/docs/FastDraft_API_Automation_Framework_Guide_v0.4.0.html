Execution contract

One request lifecycle

Folder hooks own cross-cutting preparation. Each request owns its scenario, account, explicit headers and primary response validator.

Request sandwich

Automation folder before

prepareRequestExecution() creates audit and trace state.

Foundation folder before

applyFoundationTestData() resolves common fixture defaults.

Request before

Require scenario data and call switchAccount() or the two proof rotations.

Request after

Run exactly one primary validator for the expected outcome.

Collection after

finaliseRequestExecution() completes the audit fallback.

Current collection facts

  • All 294 active requests use {{baseUrl}}; none use the retired {{server}} variable.
  • All active requests contain scalar request documentation.
  • 255 requests call switchAccount(); 2 proof requests call rotateAccounts().
  • No active request calls skipTest(). Manual skip remains available for explicit contributor use.
Order-independent preparation

Authentication and CSRF

Direct token tests validate their own response. Protected endpoint tests prepare only the selected account they need.

Account preparation flow

Resolve variables

Resolve usr_*/pwd_* or generic User/Password.

POST /Token

Retrieve bearer and secondary token fields.

Store user scope

Write auto_<userKey>_* and compatibility aliases.

GET /api/generic

Optionally retrieve CSRF state for the selected account.

Fail closed

Clear stale selected-user state when preparation fails.

Protected request example

const { switchAccount } = require("./Scripts/SystemLogic/index.js");

await switchAccount({
  usernameVar: "User"
});

// Request YAML owns these explicit headers:

// CSRF-Token: {{auto_csrfToken}}

Response contract

Response validation

Each request uses one primary validator selected by the expected outcome.

Primary validators

PrimaryReadResponseValidation

Successful controller reads, expected payload type and required/allowed/empty payload modes.

ErrorResponseValidation

Expected non-2xx JSON, text, HTML, Problem Details and allowed empty responses.

Authentication validators

Successful and failed token contracts plus successful CSRF retrieval.

Performance and payload defaults

ControlDefaultBehaviour
_responseTimeThreshold2000 msBREACH when exceeded but below the hard maximum.
_responseTimeMax30000 msFAIL when reached or exceeded.
_payloadModeallowedAllows empty or populated payload unless request-specific expectations require otherwise.
Request convention: optional performance overrides remain commented out in request scripts unless a scenario intentionally changes them.
Portal data source

Logging and reporting

The framework produces two distinct evidence families: runtime API automation logs and framework build reports.

Evidence locations

EvidenceLocationPortal ingestion approach
Runtime auditLogs/DD-MM-YY/AuditLog.txtStore raw lines plus normalized event fields.
Runtime test auditLogs/DD-MM-YY/TestAuditLog.txtUse Test Case, event ID, result, status and timing fields.
Runtime breaches/skipsBreachLog.txt / SkippedLog.txtIngest as specialised event streams.
Framework reportReports/FrameworkValidation/<run-id>/framework-build-report.jsonAuthoritative build-run document; preserve JSON and normalized summary rows.
Framework build logsLogs/FrameworkValidation/<run-id>/*.logStore file metadata and optionally parse stage output.
Latest pointerReports/FrameworkValidation/latest/ and report-index.jsonUse for discovery, not as historical identity.

Required historical data model

  • Preserve immutable source file path, hash, size and modified time.
  • Use framework run.id as the build-run identity.
  • Use runtime run ID, trace ID, Test Case ID and event ID for API event correlation.
  • Keep the original raw event line alongside parsed fields so future parsers can be improved without losing evidence.
  • Ingestion must be idempotent and must never delete source reports.

Change control

Governance and limitations

Guide content, public framework contracts, Swagger coverage and Portal ingestion mappings must be versioned together.

Approved current rules

  • Canonical request identity begins at the first [TC- marker; display icons do not change identity.
  • Manual skip syntax is await skipTest({ skipReason: "Blocked by defect FD-1234" }); as the first request pre-script call.
  • Azure Key Vault is implemented but paused by default.
  • Request-level docs contain request-specific content; folder/controller/collection docs own broader context.
  • New Portal changes are recorded in a persistent change request log with status, decisions and validation evidence.

Known current boundaries

API behaviour: HEAD, TRACE and CONNECT negative method cases remain deferred because the current API does not consistently return the intended method-not-allowed contract.
Portal baseline: the first release does not execute tests or modify Docker resources; it proves deployment and source presentation first.
